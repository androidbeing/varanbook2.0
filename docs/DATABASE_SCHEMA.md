## Multi-Tenancy & Row-Level Security

Varanbook uses **PostgreSQL RLS** to enforce tenant isolation at the database level:

| Aspect | Detail |
|--------|--------|
| **Mechanism** | Each request sets `SET LOCAL app.current_tenant_id = '<uuid>'` on the session |
| **RLS Policy** | `USING (tenant_id = current_setting('app.current_tenant_id')::uuid)` |
| **RLS-Protected Tables** | `users`, `profiles`, `shortlists`, `partner_preferences`, `file_records`, `member_subscriptions`, `tenant_plan_overrides` |
| **Non-RLS Tables** | `tenants` (public registry), `audit_logs` (cross-tenant queries by super-admins), `membership_plan_templates` (global) |
| **Super-Admin Bypass** | SUPER_ADMIN users bypass RLS via the PostgreSQL superuser role or a role with `BYPASSRLS` |
| **Denormalised tenant_id** | Tables like `profiles`, `shortlists`, and `partner_preferences` carry their own `tenant_id` to avoid costly JOINs in RLS checks |

## Common Conventions

| Convention | Description |
|------------|-------------|
| **Primary Keys** | All tables use `UUID v4` (PostgreSQL `uuid` type, generated by `uuid.uuid4()` in Python) |
| **Timestamps** | `created_at` (server-default `now()`) and `updated_at` (server-default `now()`, auto-set `onupdate`) on most tables. All `TIMESTAMPTZ` (timezone-aware) |
| **Soft Delete** | Not used globally — `is_active` flags exist on `users` and `tenants` for deactivation. Refresh tokens use `is_revoked` |
| **Cascade** | Foreign keys use `ON DELETE CASCADE` by default; `ON DELETE SET NULL` for audit-friendly references (e.g. `uploaded_by`), `ON DELETE RESTRICT` for membership plan templates |
| **Enum Storage** | Python `str` enums with `create_type=False` — values stored as plain `VARCHAR`, validated at the application/Pydantic layer |

**Design Note:** `price_paid_inr` is a **point-in-time snapshot**, not a reference to the current template price. This ensures billing history remains accurate even when plan prices are changed retroactively.

### 10. refresh_tokens

> **Model:** `app.models.refresh_token.RefreshToken`  
> **Purpose:** Stores bcrypt-hashed refresh tokens. Supports token rotation and device tracking.  
> **RLS:** Not explicitly tenant-scoped (user-scoped via `user_id`).

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | No | `uuid4()` | **PK** | Token record ID |
| `user_id` | `UUID` | No | — | **FK → users.id** (CASCADE), indexed | Owning user |
| `token_hash` | `VARCHAR(1024)` | No | — | **UNIQUE** | Bcrypt hash of the raw JWT refresh token |
| `device_info` | `VARCHAR(512)` | Yes | `NULL` | — | User-Agent / device fingerprint for UX display |
| `is_revoked` | `BOOLEAN` | No | `false` | — | Soft-revocation flag; `true` = invalidated |
| `expires_at` | `TIMESTAMPTZ` | No | — | — | Absolute expiration time |
| `created_at` | `TIMESTAMPTZ` | No | `now()` | — | When the token was issued |
| `last_used_at` | `TIMESTAMPTZ` | Yes | `NULL` | — | Last time this token was used to refresh |

**Token Lifecycle:**
```
Login           → Create new token row
/auth/refresh   → Verify hash, revoke old, create new (rotation)
/auth/logout    → Mark is_revoked = true
Nightly cleanup → Prune expired rows
```

**Security:** Raw JWTs are never stored — only bcrypt hashes. Even if the table is compromised, tokens cannot be replayed.

**Relationships:**
- `user` → `User` (many-to-one)

### 11. password_reset_tokens

> **Model:** `app.models.password_reset.PasswordResetToken`  
> **Purpose:** One-time-use password reset tokens. Only the SHA-256 hash is stored; the raw token is sent to the user's email.  
> **RLS:** Not explicitly tenant-scoped (user-scoped via `user_id`).

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | No | `uuid4()` | **PK** | Reset token record ID |
| `user_id` | `UUID` | No | — | **FK → users.id** (CASCADE), indexed | User requesting the reset |
| `token_hash` | `VARCHAR(64)` | No | — | **UNIQUE** | SHA-256 hex digest of the raw URL-safe token |
| `is_used` | `BOOLEAN` | No | `false` | — | Whether the token has been consumed |
| `expires_at` | `TIMESTAMPTZ` | No | — | — | Token expiration time |
| `created_at` | `TIMESTAMPTZ` | No | `now()` | — | When the token was generated |
| `used_at` | `TIMESTAMPTZ` | Yes | `NULL` | — | When the token was consumed |

**Reset Flow:**
```
1. POST /auth/forgot-password  → Generate raw token, store SHA-256 hash, email link
2. POST /auth/reset-password   → Verify hash, set new password, mark is_used = true
```

**Security:** SHA-256 is used (not bcrypt) because this is a short-lived, one-time-use token — not a password. The raw token is a `secrets.token_urlsafe(32)` value.

**Relationships:**
- `user` → `User` (many-to-one)
